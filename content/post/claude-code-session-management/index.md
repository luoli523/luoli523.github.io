大多数人用 Claude Code 的方式是：打开终端，开始发消息，让它一直工作。

这没有错——但如果你这么用，你其实只发挥了 Claude Code 不到一半的潜力。

Anthropic 最近公布了一套内部的 **Session 管理方法论**，来自他们与大量重度 Claude Code 用户的访谈总结。核心洞察只有一句话：**1M token 的上下文窗口是把双刃剑，不懂管理，它反而会拖垮你。**

![](intro.webp)

---

## 首先要理解：什么是 Context Rot（上下文衰退）

Claude Code 的上下文窗口有 100 万 token，听起来无限大。但有一个被称为 **Context Rot** 的现象：随着对话变长，模型性能会逐渐下降。

原因在于注意力机制——上下文越长，模型的注意力就越分散，老旧的、不相关的内容开始干扰当前任务的判断。

**Anthropic 观察到的规律**：大约在 300-400k token 时，Context Rot 开始明显出现。但这不是铁律，高度依赖任务类型——有些任务在 200k 就开始走样，有些在 600k 依然稳定。

![上下文窗口与 Context Rot 示意](context-rot.webp)

![每轮对话后的 5 个选择](session-options.webp)

![](compaction-primer.webp)

这意味着什么？**你不能因为还有余量就不管理上下文。** 就像一块硬盘，不是装满才需要整理。

## 每轮对话后，你其实有 5 个选择

绝大多数人只用了第一个：继续发消息。但实际上每当 Claude 完成一轮任务，你站在一个岔路口，有 5 条路可走：

| 操作 | 快捷键 | 适合场景 |
|------|--------|----------|
| **Continue（继续）** | 直接发消息 | 任务连续，上下文仍相关 |
| **`/rewind`（回退）** | 双击 Esc | 当前方向走错了，回到分叉点重来 |
| **`/clear`（新开）** | `/clear` | 彻底换新任务，自己写 brief |
| **`/compact`（压缩）** | `/compact` | 任务太长，上下文太重，但还要继续 |
| **Subagent（子代理）** | 指令驱动 | 独立子任务，不需要中间过程 |

这 5 个选项不是等价的，每一个背后都有不同的适用逻辑。

---

## 最被低估的技能：学会 `/rewind`

如果只能推荐一个习惯，Anthropic 的建议是：**用 `/rewind` 代替"那不对，换个方法"。**

常见场景：Claude 读了 5 个文件，尝试了方案 A，失败了。你的第一反应可能是直接说"那不行，试试方案 B"。

但这把失败的尝试留在了上下文里——Claude 读过的文件、走过的弯路、产生的中间输出，全部堆在那里，占用 token，干扰后续判断。

**更好的做法**：

1. 双击 Esc，回退到"读完文件之后"那个节点
2. 重新给指令，带上你从失败中学到的信息："不要用方案 A，foo 模块没有暴露那个接口——直接走方案 B"

你甚至可以先让 Claude 生成一份"交接摘要"——让它总结自己学到了什么，然后把这份摘要带入新的起点。

![/rewind 操作示意：回到分叉点重试，丢弃错误路径](rewind-flow.webp)

![](rewind-summary.webp)

这个思路的本质是：**保持上下文的信噪比，而不是让它无限积累。**

## `/compact` vs `/clear`：看起来像，但完全不同

当 Session 变得很长，你有两种"瘦身"方式。

**`/compact`**：让模型自己总结对话历史，用摘要替换原始记录。
- 优点：省事，Claude 可能比你总结得更全面
- 缺点：有损压缩，你得信任 Claude 的判断；可以用指令引导（`/compact 聚焦 auth 重构，丢掉测试调试部分`）

**`/clear`**：你自己写一份 brief，然后全部清空重开。
- 优点：你决定什么是重要的，上下文完全可控
- 缺点：需要你自己动脑总结
- 适合写法：「我们正在重构 auth 中间件，约束是 X，关键文件是 A 和 B，已经排除了方案 Y」

两者没有绝对优劣，关键在于**你愿意花多少精力换多少控制权**。

![/compact 压缩历史 vs /clear 全部清空重写 brief](compact-vs-clear.webp)

## 什么时候开新 Session？

Anthropic 的基本原则：**新任务 = 新 Session。**

但有个灰色地带：任务相关但不完全相同时怎么办？

举个例子：你刚实现了一个功能，接下来要写它的文档。开新 Session，Claude 要重新读一遍那些文件；继续旧 Session，上下文虽然重但省去了重读的时间。

他们的判断标准是：**这个任务对"智能强度"的要求高不高？**

写文档是相对机械的任务，多一点 Context Rot 影响不大，效率增益更重要——那就继续旧 Session。如果是需要高度精准判断的任务，宁可多花一点时间重新 load，也要保证上下文干净。

## Subagent：把独立任务"外包"出去

当你明确知道某个子任务会产生大量中间输出、而你只需要最终结果时，可以让 Claude 启动一个 **Subagent**。

Subagent 有自己独立的上下文窗口，它在那个干净的环境里工作，完成后只把结论返回给你。

判断要不要用 Subagent，只问一句：**"我需要这个过程，还是只需要结论？"**

几个典型用法：
- 「新开一个子代理，根据 spec 文件验证这次实现是否正确」
- 「新开一个子代理，读完那个代码库，总结它的 auth 实现方式，然后你自己按同样方式实现」
- 「新开一个子代理，根据 git diff 为这个功能写文档」

## 关于 Autocompact 的一个坑

如果你经常跑长 Session，可能遇到过"compact 之后感觉 Claude 变蠢了"的情况。

原因找到了：**Autocompact 触发时，模型正处于 Context Rot 最严重的时刻**——它用状态最差的自己来总结整段历史，结果自然不理想。

而且 Autocompact 会根据 Session 的主题来决定保留什么。如果你的 Session 一直在调试，然后突然切到"修那个 bar.ts 里的 warning"，那个 warning 很可能已经被当成不重要的内容丢掉了。

![Autocompact 触发时机与「坏 compact」成因](bad-compact.webp)

**解决方案**：在 Context Rot 出现之前，主动执行 `/compact`，并附上你接下来要做的事情作为引导。1M 的上下文给了你更多的提前操作空间，用好它。

## 总结：一张决策图

每次 Claude 完成一轮任务后，停下来想一想：

```
任务完成了吗？
├── 是 → 接下来是全新任务？
│         ├── 是 → /clear 新开，写好 brief
│         └── 否 → 任务相关，继续（但注意 context 长度）
│
└── 否 → 方向走错了？
          ├── 是 → /rewind 回到分叉点，重新给指令
          └── 否 → 上下文太重了？
                    ├── 是 → /compact（主动触发）
                    └── 子任务独立？ → 用 Subagent
```

1M 上下文不是用来无限堆砌的，它是给你更多主动管理的余地。

掌握这套方法，比等下一个更强的模型发布，要实用得多。
